## 1. 核心概念清单

| 概念 | 一句话解释 |

## 2. Claude Code tool_use vs OpenAI/GLM function_calling（题目考点 1）⭐

> "Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？"

### 2.1 输出样例对比

#### Claude Code（XML 嵌入文本流）

```xml
<system>
You are a helpful agent with access to tools. Use them when needed.
</system>

<user>
What's the weather in Tokyo?
</user>

<assistant>
I'll check the weather for you.
<tool_use>
{"name": "get_weather", "input": {"location": "Tokyo"}}
</tool_use>
</assistant>

<user>
<tool_result>
{"temperature": 22, "condition": "sunny", "humidity": 60}
</tool_result>
</user>

<assistant>
The weather in Tokyo is sunny, 22°C, with 60% humidity.
</assistant>
```

**关键特征**：
- 工具调用是**文本流的一部分**（被 `<tool_use>` 标签包裹）
- 可以和"思考"自然混合（"我先查一下..." + `<tool_use>`）
- 模型自由决定"何时停、调哪个"

#### OpenAI / GLM / 豆包（JSON function_calling）

```json
// Request
{
  "model": "gpt-4",
  "messages": [
    {"role": "user", "content": "What's the weather in Tokyo?"}
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "Get current weather",
        "parameters": {
          "type": "object",
          "properties": {
            "location": {"type": "string"}
          }
        }
      }
    }
  ]
}

// Response
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [
        {
          "id": "call_abc123",
          "type": "function",
          "function": {
            "name": "get_weather",
            "arguments": "{\"location\": \"Tokyo\"}"
          }
        }
      ]
    }
  }]
}
```

**关键特征**：
- 工具调用是**独立的 JSON 字段**（`tool_calls` 数组）
- `content` 通常为 `null`（纯工具调用时）
- 必须严格匹配 schema

### 2.2 详细对比表

| 维度 | Claude Code (XML) | OpenAI / GLM (JSON) |
|------|-------------------|---------------------|
| **格式** | XML 嵌入文本 | 独立 JSON 字段 |
| **解析方式** | 正则 / XML 解析 | JSON.parse + schema 验证 |
| **灵活性** | 高（可在思考中切换）| 中（一次跳转）|
| **训练契合** | 文本原生 | 结构化输出 |
| **错误恢复** | 宽松（可补全）| 严格（必须匹配 schema）|
| **可读性** | 高（看得到思考）| 低（独立字段）|
| **调试友好** | 直观（看 log 像读文章）| 需要看 JSON 结构 |
| **性能** | 略低（解析复杂）| 略高（结构化快）|
| **与文本融合** | 高 | 低 |
| **错误处理** | 模型可"补全"错误 | 直接报错 |
| **生态** | Anthropic 系 | OpenAI + 兼容生态 |
| **实现成本** | 中（parser 复杂）| 低（JSON 标准）|

### 2.3 Claude Code 这样设计的原因

**Anthropic 的设计哲学**：
- LLM 是"会思考的智能体"，不是"调函数的执行器"
- 工具调用是"思考的延伸"，不是"控制流的跳转"
- 文本流是 LLM 的"母语"，XML 标签只是"标注"

**具体动机**：
1. **训练数据契合**：Anthropic 模型的训练数据是文本流，XML 是其中自然的一部分
2. **思考-行动融合**：模型可以在"我要查 X"和"<tool_call>"之间无缝切换
3. **错误恢复**：如果 XML 标签格式不对，模型可以"继续写"补全
4. **可读性**：log 文件读起来像对话
5. **长链任务友好**：连续多步调用不会"断裂"

### 2.4 OpenAI / GLM / 豆包沿用 JSON 风格的原因

**OpenAI 的设计哲学**：
- LLM 是"调用者"，工具是"被调用的 API"
- 结构化输出 = 强约束 = 工业级可靠
- 一次跳转 = 简单控制流

**具体动机**：
1. **工程化好做**：JSON 是 Web 标准，parser 现成
2. **解析稳定**：schema 强约束，错误有明确反馈
3. **生态成熟**：函数调用、agent 框架、IDE 插件都按这个规范
4. **短链路友好**：单次调用的可靠性高
5. **高并发友好**：JSON 解析比 XML 解析快 30%+
6. **下游工具集成**：开发者写工具时，参数 schema 强校验

**GLM / 豆包沿用**：
- 兼容 OpenAI 生态（用户零迁移成本）
- 国内开发者已习惯 OpenAI 规范
- 智谱、豆包、DeepSeek 等都走 OpenAI-compatible 路线

### 2.5 OpenAI 的妥协与演进

**2023-06**：首次发布 function_calling
**2023-11**：加 `tool_choice`（强制 / 禁止调工具）
**2024-04**：加 `parallel_tool_calls`（一次调多个工具）
**2024-08**：Structured Outputs（严格 JSON schema 验证）
**2024-12**：o1 / o3 模型 + tool use 微调

**演进方向**：
- 越来越"结构化"
- 越来越"严格"
- 但本质上还是 JSON 分离式

### 2.6 实战建议

**Vibe coding 用哪个？**
- 题目要求"从零实现 Agent Runtime"
- **推荐用原生 function calling**（minimax M3 兼容 OpenAI）
- 理由：解析简单、稳定性高、符合"工程化"思路
- 如果用 Claude Code 风格：parser 复杂，但更"自然"

**什么时候用 XML 风格？**
- 想让 LLM 自由决策、文本流融合
- 学术研究、agent 自治场景
- Anthropic 模型原生支持

## 3. OpenHands 状态机设计（题目考点 2）⭐

> "Openhands的状态机设计有什么优缺？更优雅的实现方式是怎么样的？"

### 3.1 OpenHands 状态机详解

**核心设计**（基于公开源码分析）：
- 单 session 单 agent 实例
- 同步事件循环
- 状态：`IDLE / RUNNING / WAITING_USER / WAITING_TOOL / PAUSED / ERROR`
- 事件：`USER_MSG / TOOL_RESULT / TIMEOUT / USER_INTERRUPT / AGENT_DONE`

**状态转换图**：
```
                ┌────────┐
                │  IDLE  │
                └───┬────┘
                    │ user_msg
                    ↓
                ┌─────────┐
                │ RUNNING │
                └────┬────┘
            ┌────────┼────────┐
            │        │        │
       tool_call   done   timeout
            ↓        ↓        ↓
   ┌─────────────┐  ┌────┐  ┌──────┐
   │WAITING_TOOL │  │IDLE│  │ERROR │
   └──────┬──────┘  └────┘  └──────┘
          │ tool_result
          ↓
   ┌─────────┐
   │ RUNNING │  (继续)
   └─────────┘

   任意状态 + interrupt → PAUSED → IDLE
```

**核心代码逻辑**（伪）：
```python
class OpenHandsRuntime:
    def __init__(self):
        self.state = State.IDLE
        self.event_queue = Queue()
    
    def run(self):
        while True:
            event = self.event_queue.get()
            
            if self.state == State.IDLE and isinstance(event, UserMessage):
                self.state = State.RUNNING
                self.process_user_message(event)
            
            elif self.state == State.RUNNING and isinstance(event, ToolCall):
                self.state = State.WAITING_TOOL
                self.execute_tool(event)
            
            elif self.state == State.WAITING_TOOL and isinstance(event, ToolResult):
                self.state = State.RUNNING
                self.process_tool_result(event)
            
            elif isinstance(event, InterruptEvent):
                self.state = State.PAUSED
                self.save_checkpoint()
                self.state = State.IDLE
```

### 3.2 优点

| 优点 | 说明 |
|------|------|
| **清晰易调试** | 状态明确，事件明确，单线程 |
| **状态确定性强** | 同一时刻只有一个状态 |
| **容易推理** | 状态转换规则写在 runtime 里 |
| **同步简单** | 不需要处理并发、竞态 |
| **单 agent 场景高效** | 任务直接、不需要协调 |

### 3.3 缺点

| 缺点 | 说明 |
|------|------|
| **状态爆炸** | 加新功能要加新状态 + 新事件 |
| **不支持并发** | 同 session 内不能多任务并行 |
| **扩展性差** | 状态机写死，新交互模式要改核心 |
| **取消/恢复语义弱** | interrupt 后难以恢复 |
| **状态和事件耦合** | 加事件要改所有 handler |
| **分布式困难** | 状态在内存，难跨机器 |
| **难以复用** | 类似需求要重写状态机 |

### 3.4 更优雅的实现方案

#### 方案 1：Actor Model（Erlang 风格）⭐⭐

**核心思想**：每个 agent / 任务是一个 actor，消息驱动。

```python
class SessionActor:
    def __init__(self, session_id):
        self.session_id = session_id
        self.mailbox = asyncio.Queue()
        self.state = {}
    
    async def send(self, message):
        """外部 send 消息到 actor"""
        await self.mailbox.put(message)
    
    async def run(self):
        """主循环：串行处理消息"""
        while True:
            msg = await self.mailbox.get()
            await self._handle(msg)
    
    async def _handle(self, msg):
        if isinstance(msg, UserMessage):
            await self._handle_user(msg)
        elif isinstance(msg, ToolComplete):
            await self._handle_tool_complete(msg)
        # ...
```

**优点**：
- ✅ 天然支持并发（每个 actor 独立）
- ✅ 天然支持分布式（actor 可以在不同机器）
- ✅ 没有共享状态
- ✅ 失败隔离（一个 actor 崩溃不影响其他）

**缺点**：
- ❌ 调试稍复杂
- ❌ 需要消息序列化

**代表**：Akka（Java/Scala）、Elixir、Erlang OTP

#### 方案 2：事件溯源 (Event Sourcing) ⭐⭐

**核心思想**：状态不存储，只存储事件流；状态可从事件推导。

```python
# 事件流
events = [
    SessionCreated(session_id, user_id, timestamp=...),
    UserMessageReceived(content="...", timestamp=...),
    ToolCallStarted(tool="search", args=..., timestamp=...),
    ToolCallCompleted(result=..., timestamp=...),
    AgentMessageSent(content="...", timestamp=...),
    SessionPaused(timestamp=...),
    SessionResumed(timestamp=...),
]

# 当前状态 = 事件流的 fold
def replay_events(events):
    state = {}
    for event in events:
        state = event.apply(state)
    return state
```

**优点**：
- ✅ 天然 audit log
- ✅ 时间旅行调试（回到任意时刻）
- ✅ 天然支持 event sourcing + CQRS
- ✅ 持久化简单（只追加）

**缺点**：
- ❌ 状态推导慢（要 replay）
- ❌ 需要 snapshot 优化
- ❌ 模式演进困难

**代表**：Kafka + CQRS、EventStoreDB

#### 方案 3：协程 + 异步消息（Python 风格）⭐

**核心思想**：用 `asyncio` 的协程 + Queue，单线程内并发。

```python
class Session:
    def __init__(self, session_id):
        self.session_id = session_id
        self.event_queue = asyncio.Queue()
        self.running_task = None
    
    async def start(self):
        self.running_task = asyncio.create_task(self._main_loop())
    
    async def _main_loop(self):
        while True:
            event = await self.event_queue.get()
            # 处理各种事件
            await self._dispatch(event)
    
    async def send_event(self, event):
        await self.event_queue.put(event)
```

**优点**：
- ✅ 简单、Pythonic
- ✅ 单线程内并发
- ✅ 资源占用低

**缺点**：
- ❌ 不能跨进程
- ❌ 不能跨机器

**代表**：Python asyncio、Node.js

#### 方案 4：Saga Pattern ⭐

**核心思想**：长事务分解为可补偿的子事务。

```python
class Saga:
    def __init__(self, steps):
        self.steps = steps  # [step1, step2, step3]
        self.completed = []
    
    async def execute(self):
        for step in self.steps:
            try:
                await step.do()
                self.completed.append(step)
            except Exception as e:
                # 补偿
                for done in reversed(self.completed):
                    await done.undo()
                raise
```

**优点**：
- ✅ 适合长任务
- ✅ 失败可回滚

**缺点**：
- ❌ 补偿逻辑复杂
- ❌ 适合金融/订单场景，不太适合 agent

#### 方案 5：Continuation-based Runtime ⭐

**核心思想**：把状态做成闭包，可暂停可恢复。

```python
def agent_loop(session):
    while True:
        msg = yield  # 暂停点
        response = llm.chat(msg)
        if response.has_tool_call():
            result = yield tool_call(response.tool_call)
            # 继续...
        else:
            yield final_answer(response)
```

**优点**：
- ✅ 天然的暂停/恢复
- ✅ 代码即状态

**缺点**：
- ❌ 调试困难
- ❌ 不直观

**代表**：Scheme call/cc、Stackless Python、Generator-based coroutine

### 3.5 方案对比表

| 方案 | 并发 | 可恢复 | 分布式 | 复杂度 | 适合场景 |
|------|------|--------|--------|--------|----------|
| **状态机 (OpenHands)** | ❌ | 部分 | ❌ | 低 | 单机 agent、调试友好 |
| **Actor Model** | ✅ | ✅ | ✅ | 中 | 多 agent 协作、分布式 |
| **Event Sourcing** | ✅ | ✅ | ✅ | 高 | 审计要求、时间旅行 |
| **协程 (asyncio)** | ✅ | ❌ | ❌ | 低 | 单机 Python、轻量级 |
| **Saga** | ❌ | ✅ | 部分 | 中 | 长事务、金融场景 |
| **Continuation** | ❌ | ✅ | ❌ | 中 | 学术、DSL |

### 3.6 推荐选择

**Vibe coding 场景**：
- 单机、单 agent → **协程**（asyncio）
- 调试优先 → **状态机**（OpenHands 风格）

**生产环境**：
- 多 agent → **Actor Model**
- 审计要求高 → **Event Sourcing**
- 长期任务 → **Saga**
