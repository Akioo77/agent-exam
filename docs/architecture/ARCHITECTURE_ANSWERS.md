## 模块一 · Context 压缩

**题：** 一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

**答：**

关于压缩的部分先分成三层，分为短期记忆、中期记忆和长期记忆：

- **短期记忆**通过滚动对话，从当前上下文的对话中获得。
- **从短期记忆变成中期记忆**，可能通过几个方式，包括固定设计一个时间执行压缩、上下文快满了自动压缩、完成一个任务进行压缩。压缩时保留对话中的几类主要信息，如**用户偏好、任务进度、决策习惯、提到的实体**等等。中期的记忆比起长期记忆可以多详细一点，因为用户使用的概率相对长期记忆会更高。
- **中期记忆过一段固定时间进一步压缩保存为长期记忆**，进一步提取重点，节省存储空间，等待用户召回使用。

在上下文满了之后执行第一次压缩的时候，主要是省略掉对任务帮助不大的东西，同时把当前滚动记忆进行任务拆分分别保存，**同时引入任务卡概括任务主要信息**（如进度等），保留滚动记忆确保之后对话流畅。然后等待下一次对话读取到相应任务卡的记忆，保证下次继续任务时上下文有足够的容量，并且对任务的信息丢失较少。

## 模块二 · Memory 经典框架与发展趋势

**题：** 你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

**答：**

记忆的经典框架分为三层（与模块一对应）：

- **短期**：滑动窗口的记忆
- **中期**：经过压缩的记忆
- **长期**：通过向量化存入向量库

我认为的发展趋势是**把 memory 作为独立服务迭代更新**，从刚开始的只有短期记忆，发展到现在的按照需要召回中长记忆。

头部的 agent 我了解了几个：

- **MemGPT**：通过调用 API 能力，让大模型自己进行记忆管理、记忆迁移。
- **OpenAI**：偏向 RAG 路线，搜索文件 + 记忆。

## 模块三 · 长程任务的目标保持

**题：** 对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

**答：**

### 方案一：任务卡机制

在任务进程进行时，把目标、进度和约束等写成**结构化卡片**，在过程中时刻更新，并在进行任务时读取。

- **优点**：可以复用进度不会丢
- **缺点**：需要设计好 schema

### 方案二：周期性自检

可以在每 N 步让大模型**自检任务**，确保任务还在正常进行中没有出现错误（有点像 ReAct 架构）。

- **优点**：可以动态纠偏
- **缺点**：会有额外 token 花销

## 模块四 · 异步工具与完成通知

**题：** Agent 工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

**答：**

异步工具的核心思路是为每一个用户需求创造一个 **id**，然后 agent 在不影响主要进程的前提下后台执行，完成后根据 id 返回执行结果。

### 三种通知方式

1. **轮询**：用户主动询问
2. **主动推送**：agent 在 task 完成后推送到用户的对话框
3. **Webhook**：在主动推送的基础上，将执行结果推送到指定的 URL 里，让用户知道

### 用户交互设计

在异步工具使用时候应该注重和用户的交互，比如在 **busy 状态**时让用户知道，告诉用户"正在处理中"。

我会设计成向用户发送：

- **task_id**（用于后续查询）
- **预计完成时间**（让用户有预期）

完成时**主动将结果 push 到用户能收到的地方**，达到效率最大化。

## 模块五 · Claude Code vs GLM/豆包 工具输出

**题：** Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

**答：**

### 设计差异

- **Claude Code**：工具输出是 **XML 嵌入文本式**
- **GLM / 豆包等**：使用 **JSON** 结构化输出

### Claude Code（XML）的优劣

**优点：**
- XML 比起 JSON 更加灵活
- 可以将思考和行动相结合，工具调用穿插在文字里
- 更加灵活的输出可以让大模型有**修补错误的空间**，没有将工具写死
- 调试看日志时文件更具有**可读性**

**缺点：**
- 更加灵活的文本使得大模型处理相关工作时比起结构固定的 JSON 会**花费较多时间**

### GLM / 豆包（JSON）的优劣

**优点：**
- **兼容性强**
- **工程化简单**
- **约束性强**
- 在工业生产环境时更加**稳定可靠**

**缺点：**
- 不灵活
- 可读性差

## 设计哲学总结

| 维度 | Claude Code（XML）| GLM/豆包（JSON）|
|---|---|---|
| 本质 | 文本原生，工具是思考的延伸 | API 调用，工具是结构化函数 |
| 哲学 | LLM 是会思考的智能体 | LLM 是函数调用器 |
| 优势 | 灵活、可读、可恢复 | 稳定、兼容、约束强 |
| 适用 | 学术研究、长链任务 | 工业生产、高并发 |
